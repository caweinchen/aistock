# 普通用户可信度与留存增强设计

## 背景

普通用户 MVP 已完成克制参考状态、自选股分组、普通用户摘要、数据健康字段和重要股东文案调整。下一步不进入会员、支付、专业版或高级策略，而是把现有 MVP 打磨成更可信、更容易持续使用的普通用户分析体验。

长期产品定位是“股票学习、风险识别、自选股分析与决策辅助平台”，不是荐股工具。本阶段继续坚持这个边界：系统提供学习和风险检查，不替用户做交易决定。

## 目标

1. 让用户清楚知道每个结论基于什么数据，以及数据是否足够可信。
2. 让用户理解一只股票的主要风险来自哪里，而不是只看到一个状态或分数。
3. 让用户在决策前回到产品完成检查清单，提高留存和复访理由。
4. 保持现有 API 兼容，继续复用本地数据、因子、风险提示、历史行情、重要股东和分红数据。
5. 为后续普通会员能力预留清晰边界，但本阶段不实现商业化权限。

## 非目标

- 不做会员、订阅、支付或价格方案。
- 不做管理后台、报告导出、批量分析或专业版能力。
- 不做完整持仓仓位管理。
- 不输出“立即买入”“强烈卖出”“必涨”“稳赚”“最佳买点”等交易指令。
- 不把回测结果包装成未来收益承诺。

## 产品范围

本阶段包含四个模块：

1. MVP 验收闭环
2. 数据健康中心 v1
3. 个股风险说明书 v1
4. 操作前检查清单 v1

### MVP 验收闭环

在新增功能前，先完成当前 MVP 计划中未完成的验证：

- 后端全量测试通过。
- `/api/watchlist/insights` 在真实登录 token 下可返回自选股分组。
- 首页能展示“今日自选股参考”。
- 个股详情能展示普通用户摘要、数据健康、支撑因素、风险因素。
- 页面中不出现直接交易指令类文案。

验收闭环的产出是一个明确的验证记录，写回现有 MVP 计划或新增验证文档。

### 数据健康中心 v1

数据健康中心不是单独的大页面，先作为首页自选股洞察和个股详情中的统一信息块出现。

展示内容：

- 数据完整度：完整、基本完整、不完整、数据不足。
- 最近更新时间。
- 缺失数据项。
- 结论降级原因。
- 数据来源摘要：EastMoney、TuShare Pro、本地缓存、本地计算。

用户看到“数据不足”时，应知道为什么不足，例如历史行情少、财务因子缺失、重要股东数据缺失、风险提示无法形成判断。

后端增加统一的数据健康结构，避免每个接口临时拼文案。

建议响应模型：

```text
DataHealth {
  completeness: complete | mostly_complete | incomplete | insufficient
  updated_at: datetime | null
  source_summary: string[]
  missing_items: string[]
  downgrade_reasons: string[]
  user_message: string
}
```

现有 `data_completeness`、`data_updated_at` 字段保留，新增 `data_health` 作为更完整结构。前端优先读取 `data_health`，没有时回退旧字段。

### 个股风险说明书 v1

风险说明书用于解释“为什么要谨慎”，不用于输出买卖建议。

风险类型：

- 估值风险
- 波动风险
- 业绩或因子风险
- 重要股东变动风险
- 分红稳定性风险
- 数据不足风险

每个风险项包含：

```text
RiskExplanation {
  type: valuation | volatility | fundamentals | holder_change | dividend | data_quality
  level: low | medium | high
  title: string
  what_it_means: string
  why_it_matters: string
  evidence: string[]
}
```

示例文案：

- 风险是什么：当前估值指标偏高，价格可能已经反映较多乐观预期。
- 为什么重要：如果后续业绩增长跟不上，股价可能承压。
- 证据：估值因子低于阈值、近期涨幅较大、波动风险偏高。

风险说明书在个股详情页中放在普通用户摘要之后、专业因子和策略模块之前。这样普通用户先看结论和风险，再决定是否继续看复杂数据。

### 操作前检查清单 v1

检查清单是留存核心：用户不只是在浏览股票，而是在操作前回到产品做确认。

支持两类检查：

- 买入前检查
- 卖出前检查

买入前检查项：

- 是否了解公司主营业务。
- 当前估值是否偏高。
- 最近业绩是否稳定。
- 是否存在重要股东明显减持。
- 股价是否刚经历大幅上涨。
- 下跌 10%-20% 是否能承受。
- 是否存在单只股票或同类股票过度集中风险。

卖出前检查项：

- 买入逻辑是否已经失效。
- 业绩是否明显恶化。
- 估值是否过高。
- 重要股东是否连续减持。
- 是否只是因为短期波动而恐慌。
- 是否有更清楚的风险证据支持减仓。

检查项分为系统判断和用户确认两类：

- 系统判断：由数据和规则生成，例如估值偏高、波动偏高、重要股东减持。
- 用户确认：由用户手动勾选，例如是否了解主营业务、是否能承受回撤。

本阶段只保存本地 UI 状态，不做后端持久化。后续如果进入会员或留痕阶段，再增加审计记录。

## 后端设计

### 服务边界

继续保持轻量，不做大拆分，但在 `backend/app/main.py` 中把新增逻辑集中成清晰的辅助函数，避免散落在路由里。

建议新增函数：

- `build_data_health(stock, history, factors, alerts, holders, dividends) -> DataHealth`
- `build_risk_explanations(stock, factors, alerts, holders, dividends, data_health) -> list[RiskExplanation]`
- `build_pre_trade_checklist(stock, risk_explanations, data_health, mode) -> PreTradeChecklist`

如果 `main.py` 继续膨胀，实施计划中应考虑将普通用户决策逻辑移到独立模块，例如 `backend/app/ordinary_user.py`。该拆分只服务本阶段逻辑，不做全后端重构。

### API 变化

优先扩展现有个股详情接口：

- `GET /api/stocks/{code}` 新增 `data_health`、`risk_explanations`、`buy_checklist`、`sell_checklist`。

自选股洞察接口可新增聚合健康信息：

- `GET /api/watchlist/insights` 新增 `data_health_overview`，用于首页展示整体数据可信度。

所有新增字段应为可选或带默认值，保持旧缓存和旧前端兼容。

## 前端设计

### 首页

首页在“今日自选股参考”中增加一个轻量数据健康提示：

- 数据整体是否足够。
- 有多少只股票数据不足。
- 最近更新时间。

如果存在数据不足股票，首页应把它们明确放在“数据不足”分组，不强行给积极或谨慎结论。

### 个股详情页

详情页展示顺序：

1. 价格与普通用户摘要。
2. 数据健康。
3. 个股风险说明书。
4. 操作前检查清单入口。
5. 支撑因素和风险因素。
6. 因子、策略、分红、重要股东、重大事件等专业信息。

这样普通用户先看到可理解内容，进阶用户仍可继续查看底层数据。

### 检查清单交互

检查清单采用折叠区域或分段控件：

- 买入前
- 卖出前

每个检查项显示：

- 状态：通过、需注意、用户确认、数据不足。
- 一句解释。
- 用户可勾选的确认项。

不显示“建议买入”或“建议卖出”，只显示“检查完成度”和“仍需关注的问题”。

## 数据与规则

本阶段使用规则评分，不引入复杂模型。

规则来源：

- 因子分数：估值、动量、波动、盈利等。
- 风险提示：高、中、低风险。
- 历史行情数量和更新时间。
- 重要股东变动。
- 分红记录。
- 股票本身的 `data_status`、`updated_at`、`history_updated_at`。

数据不足优先级最高：如果关键数据不足，应先降级结论，再解释缺失项。

## 测试策略

后端测试：

- 数据健康完整、基本完整、不完整、数据不足四种场景。
- 风险说明书能根据因子和风险提示生成不同风险项。
- 数据不足时不生成过强结论。
- 个股详情新增字段兼容旧字段。
- 自选股洞察能汇总数据不足数量。

前端测试：

- API 类型包含新字段。
- 旧响应缺少 `data_health` 时能回退。
- 首页展示数据健康概览。
- 详情页展示风险说明书和检查清单。
- 文案中不出现直接交易指令。

手动验收：

- 使用真实 token 调用 `/api/watchlist/insights`。
- 打开首页确认自选股分组、数据健康和免责声明正常。
- 打开个股详情确认风险说明书与检查清单正常。
- 检查移动端窄屏文本不重叠。

## 成功指标

本阶段成功不以交易转化衡量，而看用户是否更愿意反复使用：

- 自选股洞察页打开率。
- 个股详情页停留和复访。
- 风险说明书展开率。
- 检查清单完成率。
- 数据不足提示后的刷新或补数据行为。
- 用户是否持续回来看自选股变化。

## 实施顺序

1. 完成 MVP 验收闭环。
2. 后端增加 `DataHealth` 与测试。
3. 后端增加风险说明书与测试。
4. 后端增加检查清单响应与测试。
5. 前端增加类型和 API 回退。
6. 首页展示数据健康概览。
7. 详情页展示数据健康、风险说明书和检查清单。
8. 完成后端、前端、TypeScript 和手动冒烟验证。

## 风险与取舍

- 如果规则过复杂，用户会看不懂；本阶段规则应少而清楚。
- 如果只堆更多数据，留存不会明显改善；必须把数据转成普通语言解释。
- 如果过早做会员，会在核心价值未验证前增加系统复杂度。
- 如果不做数据不足降级，产品可信度会受损。

本阶段的判断标准是：用户能不能清楚理解“为什么系统这么说”，以及是否愿意在决策前回来做一次检查。
