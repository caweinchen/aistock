# 阶段联调门禁设计

## 目标

两台电脑完成任一阶段性变更后，必须先完成前后端联调并记录证据；只有联调通过，下一阶段才能进入执行状态。

## 适用范围

本门禁适用于会影响前后端协作结果的阶段，包括后端接口与数据语义变更、前端接口接入与关键用户路径变更，以及跨端缺陷修复。纯文档整理如果不改变规格、契约或运行行为，可以不启动联调，但仍需完成文档检查并记录豁免原因。

## 状态流转

每个阶段按以下状态流转：

1. `In Progress`：当前阶段正在开发。
2. `Awaiting Integration`：开发与本机验证已完成，等待两端基于同一 `main` 提交联调。
3. `Integration Passed`：真实前后端关键路径验证通过，证据已写回文档并推送。
4. `Completed`：阶段正式完成，允许下一阶段进入 `In Progress`。

联调失败时，阶段回到 `In Progress` 或保持 `Awaiting Integration`，并新增带 Owner、原因和验收条件的显式 TODO。不得将下一阶段改为 `In Progress`。

## 门禁通过条件

联调必须同时满足：

- 前后端工作机已同步到看板记录的同一远端 `main` 基准。
- 后端服务和必要依赖可以按现有联调说明启动。
- 前端访问真实接口，不以 mock 响应代替联调结论。
- 当前阶段规格或契约涉及的关键路径已验证。
- 至少验证成功路径；涉及鉴权、空态或错误语义时，也验证对应失败路径。
- 前端和后端各自要求的自动化验证通过。
- 联调结果、命令或操作、基准提交和阻塞项已写回计划或看板。
- 文档更新已提交并推送到远端 `main`。

只有以上条件全部满足，联调 TODO 才能标为 `[x]`。

## 两机职责

后端电脑负责后端启动、数据库与迁移状态、后端测试、接口契约和服务端问题修复。前端电脑负责前端启动、真实接口接入、前端测试、关键用户路径和客户端问题修复。

任一电脑发现职责范围外的问题时，只记录缺陷和证据并指定正确 Owner；未经用户明确授权，不跨越角色边界修改代码。

## 看板要求

每个阶段至少记录：

- 阶段名称和 Owner。
- 当前状态。
- 前端基准提交与后端基准提交。
- 联调计划或验证记录路径。
- 自动化验证结果。
- 手工关键路径结果。
- 阻塞项及其 Owner。
- 下一阶段；门禁通过前必须保持 `Blocked by integration gate` 或 `Pending`。

## 异常与豁免

- 联调环境不可用：记录环境故障、负责人和恢复条件，当前阶段不得完成。
- 契约与实现不一致：先修正规格、契约或实现，再重新联调。
- 纯文档变更：只有确认不改变规格、契约或运行行为时可以豁免联调，并在提交或看板写明原因。
- 紧急绕过：仅用户明确授权时允许，并必须留下未完成联调 TODO，后续阶段仍受该 TODO 阻塞，直到补验通过。

## 验收标准

- 协作规约包含阶段联调硬门禁和状态流转。
- 前端机与后端机角色文件均要求执行门禁。
- 项目看板提供联调门禁字段，并明确下一阶段是否可启动。
- 任一阶段未通过联调时，文档流程无法将下一阶段标为进行中。
